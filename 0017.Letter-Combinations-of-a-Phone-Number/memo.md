# 17. Letter Combinations of a Phone Number



## step1
DFSで解いた。10分ぐらい。

以下n = digits の長さ、k = 1桁あたりの最大文字数（最大4）

- 時間計算量: O(n × k^n)
  - 結果は最大 k^n 個、各結果の "".join() に O(n) かかる
  - frontier.append 時に fixed + [c] でリストコピーが発生するが同オーダー
- 空間計算量: O(n × k^n)
  - スタックに最大 k^n 個のノードが積まれ、各ノードが長さ n のリストのコピーを持つ

## step2

### 再帰 + バックトラッキング

- 時間計算量: O(n × k^n)
  - 葉ノードは k^n 個、各葉で "".join(current) に O(n) かかる
- 空間計算量: O(n)（出力除く）
  - 再帰の深さ n、current リストの長さ n のみ
  - step1 と違い current を使い回すのでコピーが発生しない

### イテレーティブ BFS

- 時間計算量: O(n × k^n)
  - 各桁ごとにキュー内の全要素に1文字連結、最終的に k^n 個の長さ n の文字列
- 空間計算量: O(n × k^n)
  - キューに最大 k^n 個の文字列が入り、各文字列の長さは最大 n
  - step1 と同様にコピーを持つので空間効率は変わらない

「文字列の追記は文字列の再構築が走るので、リストに append して join するのが良い」理由を整理:

Pythonの文字列がimmutableなので、文字列の再構築を行うとメモリ領域に新しく文字列を確保する必要があり、文字を追加するたびに O(n) の計算量がかかるため

## バックトラック vs DFS/BFS

| | バックトラック（再帰） | イテレーティブ DFS/BFS |
|---|---|---|
| **空間効率** | O(n)（状態を使い回す） | O(n × k^n)（各ノードがコピーを持つ） |
| **枝刈り** | しやすい（再帰の条件分岐で自然に書ける） | できるが記述が増える |
| **コードの見通し** | 問題の構造と対応しやすい | スタック/キューの管理が必要 |
| **スタックオーバーフロー** | 再帰が深いと起きうる | 起きない |
| **適した問題** | 制約付き全列挙（Subsets, Permutations 等） | グラフ探索、最短経路（BFS） |

-> 全列挙・組み合わせ系はバックトラックが自然。BFS は「最短」を求めるときに真価を発揮する。
